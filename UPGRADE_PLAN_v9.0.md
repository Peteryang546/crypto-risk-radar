# Crypto Risk Radar v9.0 升级计划

## 升级目标

基于"区块链币调研再次升级"需求，将 Crypto Risk Radar 从 v8.7 升级到 v9.0，实现：
1. 每日报告增加"链上异常事实清单"模块
2. 建立代币深度研究框架（五大软肋检测）
3. 设计人机合作流程
4. 添加留言板块（接受公众批评）
5. 确保内容公开可共享

---

## 一、每日报告升级（Module 14）

### 新增模块：链上异常事实清单

**位置**: Module 14（在 Risk Matrix 之后）

**格式**: 对比表格

| 列 | 内容 |
|---|---|
| 代币 | 代币符号 + 合约地址（缩短） |
| 项目方/KOL声称 | 从官网/推文提取的声明 |
| 链上/公开事实 | 实际检测到的数据 |
| 来源/验证方式 | 可点击的链接 |

**数据来源**:
- GoPlus Security API（合约安全）
- DEX Screener（流动性、持仓）
- Etherscan（交易历史）
- 社交媒体（KOL推文）

**筛选条件**（满足任一即列入）:
- 流动性 < $50k
- 前10地址持仓 > 50%
- GoPlus 检测出高危特征（honeypot、隐藏所有者等）
- 解锁事件即将发生

**风格要求**:
- 不写结论
- 不使用"建议""不要"等词
- 只陈列事实对比

## 二、代币深度研究框架

### 2.1 五大软肋检测表

| 软肋类别 | 检测项 | 数据来源 |
|---|---|---|
| **合约代码风险** | Honeypot检测 | GoPlus API |
| | 卖出税率 | GoPlus API |
| | 隐藏所有者 | GoPlus API |
| | 合约是否开源 | Etherscan |
| **持仓分布** | 前10地址持仓比例 | DEX Screener |
| | 前50地址持仓比例 | DEX Screener |
| | 新增钱包数量（24h） | Etherscan |
| **流动性管理** | LP是否锁仓 | DEX Screener |
| | 锁仓时长 | Unicrypt |
| | 流动性规模 | DEX Screener |
| **开发者关联** | 团队是否实名 | 官网/LinkedIn |
| | 历史关联地址 | Arkham |
| | 开发者钱包活动 | Etherscan |
| **营销叙事** | 承诺高回报 | 官网/KOL推文 |
| | 叙事与事实矛盾 | 对比检测 |

### 2.2 代币分类体系

| 类型 | 定义 | 典型信号 | 颜色标记 |
|---|---|---|---|
| 🔴 极高风险（骗局） | 明确欺诈性合约或撤池行为 | honeypot、流动性未锁、内部钱包提前出货 | #ff4444 |
| 🟠 高风险（可疑） | 项目方不透明，叙事与事实严重矛盾 | 团队匿名、持仓高度集中、KOL喊单前内部买入 | #ff8800 |
| 🟡 中风险（经营困难） | 有真实产品但财务/运营不稳定 | 资金快耗尽、开发停滞、社区萎缩 | #ffcc00 |
| 🟢 低风险（可研究） | 运行多年，信息透明，网络健壮 | BTC、ETH等 | #44ff44 |

### 2.3 应对策略（供读者选择）

| 类型 | 策略描述 |
|---|---|
| 🔴 极高风险 | 直接忽略。不建议任何形式的参与。 |
| 🟠 高风险 | 远离。除非你是专业赌徒，否则不要参与。 |
| 🟡 中风险 | 观察，不参与。等待项目方公布解决方案或自然消亡。 |
| 🟢 低风险 | 可学习、可研究。作为理解区块链技术的参考，不构成投资建议。 |

## 三、人机合作流程设计

### 3.1 流程概述

```
[自动发现] → [自动扫描] → [人工复核] → [分类决策] → [发布]
     ↑                                              ↓
     └──────────── [反馈学习] ←─────────────────────┘
```

### 3.2 具体步骤

**步骤1: 自动发现**
- 每日自动扫描 DEX Screener 新代币
- 筛选条件：流动性 < $100k 或 24h 交易量 > $10k
- 输出：候选代币列表（JSON）

**步骤2: 自动扫描**
- 对候选代币执行五大软肋检测
- 输出：检测报告（Markdown表格）

**步骤3: 人工复核**
- 用户查看检测报告
- 判断是否需要深入研究
- 决策：跳过 / 深入研究 / 立即分类

**步骤4: 分类决策**
- 用户根据检测结果和主观判断，确定代币类型
- 选择应对策略
- 添加备注（可选）

**步骤5: 发布**
- 自动更新每日报告（链上异常事实清单）
- 自动更新代币分类总表
- 如为骗局，添加至案例库

**步骤6: 反馈学习**
- 跟踪已分类代币的后续发展
- 验证分类准确性
- 调整检测阈值和权重

### 3.3 用户交互界面

**待办列表**（每日生成）:

## 四、留言板块设计

### 4.1 技术方案

**方案A: GitHub Issues**（推荐）
- 每个报告页面底部添加"评论"按钮
- 链接到 GitHub Issues，带预填充标签
- 优点：免费、永久保存、可追踪
- 缺点：需要GitHub账号

**方案B: Disqus**（备选）
- 嵌入式评论系统
- 优点：用户友好，无需GitHub
- 缺点：免费版有广告，数据不自主

**方案C: 静态评论**（极简）
- 使用 GitHub Actions + GitHub Issues API
- 定期拉取评论，生成静态评论页面
- 优点：完全静态，加载快
- 缺点：评论非实时

### 4.2 推荐实现（方案A）

在报告页面底部添加：
```html
<div class="comments-section">
    <h2>💬 评论与反馈</h2>
    <p>欢迎对报告内容提出批评、建议或补充信息。</p>
    <a href="https://github.com/peteryang546/crypto-risk-radar/issues/new?labels=feedback&title=反馈：报告日期 2026-04-16" 
       target="_blank" class="comment-btn">
       在 GitHub 上提交反馈
    </a>
    <p class="comment-note">需要 GitHub 账号。如无账号，可发送邮件至 feedback@crypto-risk-radar.com</p>
</div>
```
